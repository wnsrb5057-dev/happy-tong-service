# React + PWA 개발 Rule
## 1. 프로젝트 목표
이 프로젝트는 React 기반의 PWA 웹앱을 개발하는 프로젝트입니다.
최종 목표는 사용자가 브라우저에서 접속할 수 있고, 필요할 경우 홈 화면에 앱처럼 설치할 수 있는 웹서비스를 만드는 것입니다.
AI는 기존 기능을 안전하게 유지하면서 React 컴포넌트 구조, 상태 관리, 라우팅, PWA 설정, 반응형 UI, 빌드 안정성을 고려하여 작업해야 합니다.
---
## 2. 작업 전 확인
* 사용자의 요청을 먼저 정확히 이해한 뒤 작업한다.
* 요청이 애매하면 임의로 판단하지 말고 먼저 확인한다.
* 수정 전 프로젝트 파일 구조를 확인한다.
* 기존 코드 흐름, 컴포넌트 구조, 라우팅 구조를 확인한 뒤 작업한다.
* 전체를 새로 만들기보다 필요한 부분만 수정한다.
* 대규모 변경이 필요하면 먼저 이유와 변경 범위를 설명한다.
---
## 3. 기존 기능 보호
* 기존에 정상 작동하던 기능을 함부로 삭제하지 않는다.
* 사용자가 요청하지 않은 기능을 임의로 추가하지 않는다.
* 기존 파일명, 폴더 구조, 함수명, 컴포넌트명을 마음대로 바꾸지 않는다.
* 라우팅 경로를 임의로 변경하지 않는다.
* 기존 UI/UX 흐름을 크게 바꿀 경우 먼저 설명한다.
* 기존 상태 관리 방식이 있다면 그 방식을 우선 따른다.
* 기존 데이터 구조를 변경할 경우 영향 범위를 먼저 확인한다.
---
## 4. 기술 스택 기준
* React를 중심으로 개발한다.
* 가능하면 함수형 컴포넌트를 사용한다.
* 상태 관리는 기본적으로 useState, useEffect, useMemo, useCallback 등 React 기본 훅을 우선 사용한다.
* 프로젝트가 Vite 기반이면 Vite 구조를 유지한다.
* 프로젝트가 Next.js 기반이면 Next.js 구조를 유지한다.
* TypeScript 프로젝트라면 타입을 명확히 작성한다.
* JavaScript 프로젝트라면 불필요하게 TypeScript로 변경하지 않는다.
* 사용자가 요청하지 않은 프레임워크로 변경하지 않는다.
---
## 5. React 코드 작성 규칙
* 컴포넌트는 역할별로 분리한다.
* 하나의 컴포넌트가 너무 길어지면 작은 컴포넌트로 나눈다.
* 컴포넌트 이름은 PascalCase로 작성한다.
* 변수명과 함수명은 의미 있게 작성한다.
* props 이름은 역할이 드러나게 작성한다.
* 반복되는 UI는 재사용 가능한 컴포넌트로 만든다.
* 중복 코드는 줄이되, 지나치게 복잡한 추상화는 피한다.
* 불필요한 useEffect 사용을 피한다.
* 상태 변경은 직접 수정하지 말고 setState를 사용한다.
* map 사용 시 key를 반드시 지정한다.
* 조건부 렌더링은 초보자도 이해할 수 있게 명확하게 작성한다.
* 이벤트 핸들러 함수명은 handleClick, handleSubmit처럼 의미 있게 작성한다.
---
## 6. PWA 개발 규칙
* PWA 기능을 추가할 때는 manifest 파일과 service worker 설정을 확인한다.
* 앱 이름, 아이콘, 시작 URL, display 설정, theme_color, background_color를 명확히 작성한다.
* PWA는 HTTPS 환경 또는 localhost에서 정상 동작한다는 점을 고려한다.
* service worker는 무분별하게 캐싱하지 않는다.
* HTML, JS, CSS, 이미지 등 정적 파일 캐싱 전략을 명확히 한다.
* 오래된 캐시 때문에 화면이 갱신되지 않는 문제가 생기지 않도록 버전 관리 또는 캐시 갱신 전략을 고려한다.
* 오프라인 화면이 필요한 경우 사용자에게 알기 쉬운 안내 화면을 제공한다.
* PWA 설정 변경 후에는 브라우저 개발자 도구의 Application 탭에서 manifest와 service worker 상태를 확인할 수 있게 안내한다.
* 설치 가능 여부를 확인할 수 있도록 manifest와 아이콘 경로 오류를 점검한다.
---
## 7. UI/UX 규칙
* 화면은 모바일과 데스크톱 모두에서 보기 좋게 반응형으로 작성한다.
* 버튼, 입력창, 카드 UI는 사용자가 누르기 쉽게 충분한 크기로 만든다.
* 색상은 일관성 있게 사용한다.
* 중요한 버튼과 안내 문구는 눈에 잘 보이게 작성한다.
* 빈 화면, 로딩 상태, 오류 상태를 고려한다.
* 사용자가 입력하는 폼에는 기본적인 검증을 넣는다.
* 알림은 alert만 남발하지 말고 가능하면 화면 안에서 안내한다.
* 접근성을 고려해 버튼, 입력창, 이미지에는 적절한 텍스트 정보를 제공한다.
---
## 8. 파일 구조 규칙
* 기존 프로젝트 구조를 우선 따른다.
* 새 파일을 만들 때는 역할이 명확한 위치에 만든다.
* 컴포넌트는 components 폴더에 둔다.
* 페이지 단위 파일은 pages 또는 routes 구조를 따른다.
* 공통 유틸 함수는 utils 폴더에 둔다.
* 이미지와 아이콘은 assets 또는 public 폴더에 둔다.
* PWA 아이콘, manifest, service worker 관련 파일은 public 또는 프로젝트 설정에 맞는 위치에 둔다.
* 임의로 폴더 구조를 크게 변경하지 않는다.
---
## 9. 패키지 설치 규칙
* npm install이 필요한 패키지를 임의로 추가하지 않는다.
* 패키지가 필요하면 먼저 이유를 설명한다.
* 이미 설치된 패키지로 해결 가능한지 먼저 확인한다.
* 무거운 라이브러리보다 가벼운 구현을 우선 고려한다.
* 사용하지 않는 패키지를 추가하지 않는다.
* package.json을 수정할 경우 변경 이유를 명확히 설명한다.
---
## 10. 보안 규칙
* API 키, 비밀번호, 토큰 같은 민감정보를 코드에 직접 작성하지 않는다.
* 민감정보는 환경변수로 관리한다.
* 브라우저에 노출되면 안 되는 키는 프론트엔드 코드에 넣지 않는다.
* 사용자 입력값은 가능한 한 검증한다.
* 개인정보를 console.log로 출력하지 않는다.
* localStorage에 민감한 정보를 저장하지 않는다.
* 인증, 결제, 데이터베

---

## 11. Agent 토큰 최소화 규칙

* AI Agent는 작업 범위를 불필요하게 넓히지 않는다.
* 사용자가 요청한 기능, 오류, 파일과 직접 관련된 부분부터 확인한다.
* 프로젝트 전체를 한 번에 분석하지 않는다.
* 전체 파일 구조 확인이 필요하더라도 먼저 최상위 구조만 간단히 확인한다.
* 관련 없는 폴더와 파일은 임의로 열람하지 않는다.
* 파일을 열기 전, 왜 이 파일을 확인해야 하는지 스스로 판단한다.
* 같은 파일을 반복해서 읽지 않는다.
* 이미 확인한 내용은 다시 탐색하지 않는다.
* 긴 코드 전체를 출력하지 않고, 변경된 부분만 중심으로 설명한다.
* 수정 결과 설명은 변경 파일, 변경 이유, 확인 방법 중심으로 짧게 작성한다.
* 불필요한 예시 코드, 장황한 설명, 중복 설명을 피한다.
* 사용자가 요청하지 않은 리팩토링, 구조 개선, 디자인 개선을 하지 않는다.
* 문제 해결에 꼭 필요한 경우가 아니면 새 파일을 만들지 않는다.
* 문제 해결에 꼭 필요한 경우가 아니면 새 패키지를 설치하지 않는다.
* 검색, 분석, 테스트, 수정은 필요한 최소 범위에서 수행한다.

---

## 12. Agent 작업 진행 방식

* 먼저 사용자의 요청을 한 문장으로 요약한다.
* 바로 수정하지 말고, 필요한 경우 수정 계획을 3~5줄 이내로 작성한다.
* 수정 계획에는 확인할 파일, 수정할 파일, 예상 변경 범위를 포함한다.
* 사용자가 특정 파일을 지정한 경우 해당 파일을 우선 확인한다.
* 사용자가 파일을 지정하지 않은 경우 관련성이 높은 파일부터 순서대로 찾는다.
* 처음부터 전체 프로젝트를 스캔하지 않는다.
* 오류 해결 작업에서는 에러 메시지, 발생 파일, 관련 컴포넌트만 우선 확인한다.
* 기능 추가 작업에서는 UI, 상태, API, 저장 로직을 한 번에 모두 바꾸지 않고 단계별로 진행한다.
* 한 번의 작업에서는 하나의 명확한 목표만 처리한다.
* 여러 기능이 섞인 요청은 작업 단위를 나누어 처리한다.
* 수정 후에는 전체 코드를 다시 설명하지 않고 변경 요약만 제공한다.

---

## 13. 파일 탐색 최소화 규칙

* 관련 파일을 찾을 때는 파일명, 컴포넌트명, 라우트명, 함수명 기준으로 우선 검색한다.
* `src` 전체를 무작정 읽지 않는다.
* `node_modules`, `dist`, `build`, `.next`, `.vite`, `.git` 폴더는 확인하지 않는다.
* 빌드 결과물이나 캐시 파일은 수정하지 않는다.
* package-lock.json, pnpm-lock.yaml, yarn.lock은 패키지 변경이 있을 때만 확인한다.
* README, 문서 파일은 작업에 직접 필요할 때만 확인한다.
* 이미지, 아이콘, 폰트 파일은 UI 또는 PWA 설정과 직접 관련 있을 때만 확인한다.
* 한 파일에서 필요한 함수나 컴포넌트만 확인하고, 관련 없는 영역은 분석하지 않는다.
* 파일 구조를 파악할 때는 전체 트리 대신 필요한 폴더 깊이까지만 확인한다.

---

## 14. 출력 토큰 최소화 규칙

* 답변은 짧고 명확하게 작성한다.
* 수정 전 설명은 5줄 이내로 제한한다.
* 수정 후 요약은 다음 형식을 따른다.

### 변경 파일

* `파일명`: 변경 내용 요약

### 변경 이유

* 왜 수정했는지 간단히 설명

### 확인 방법

* 실행 또는 테스트 방법만 간단히 안내

* 전체 코드를 다시 붙여넣지 않는다.

* 사용자가 요청하지 않으면 긴 설명을 하지 않는다.

* 코드 설명보다 실제 변경 결과를 우선한다.

* 같은 내용을 반복하지 않는다.

* “추가로 개선할 수 있는 점”은 사용자가 요청했을 때만 제안한다.

---

## 15. 최소 수정 원칙

* 기존 기능을 유지하면서 필요한 부분만 수정한다.
* 기존 코드 스타일을 따른다.
* 기존 컴포넌트 구조를 최대한 유지한다.
* 기존 상태 관리 방식을 유지한다.
* 기존 라우팅 구조를 유지한다.
* 기존 CSS 구조를 유지한다.
* 새 구조를 만들기보다 기존 구조 안에서 해결한다.
* 작은 오류 수정에 대규모 리팩토링을 하지 않는다.
* 단순한 문제를 복잡한 구조로 해결하지 않는다.
* 임시 해결보다 안전한 최소 해결을 우선한다.

---

## 16. Agent에게 금지할 행동

* 요청과 관련 없는 파일을 대량으로 열람하지 않는다.
* 프로젝트 전체를 임의로 리팩토링하지 않는다.
* 사용자가 요청하지 않은 기능을 추가하지 않는다.
* 사용자가 요청하지 않은 디자인 변경을 하지 않는다.
* 사용자가 요청하지 않은 패키지를 설치하지 않는다.
* 사용자가 요청하지 않은 TypeScript 전환을 하지 않는다.
* 사용자가 요청하지 않은 상태 관리 라이브러리를 추가하지 않는다.
* 기존 파일명, 폴더명, 컴포넌트명을 임의로 변경하지 않는다.
* 정상 작동하는 코드를 추측으로 삭제하지 않는다.
* 긴 분석 보고서를 작성하지 않는다.
* 수정과 무관한 학습용 설명을 길게 하지 않는다.

## 17. Codex / Agent 실행용 기본 지시문

Agent는 아래 원칙을 항상 따른다.

```text
작업 범위를 최소화한다.
관련 파일만 확인한다.
전체 프로젝트를 불필요하게 분석하지 않는다.
기존 구조를 유지한다.
최소 수정으로 해결한다.
요청하지 않은 리팩토링은 하지 않는다.
수정 전 계획은 짧게 작성한다.
수정 후 요약도 짧게 작성한다.
전체 코드 출력은 피하고 변경된 부분만 설명한다.
```

---

## 18. 권장 작업 순서

1. 요청을 한 문장으로 요약한다.
2. 관련 파일 후보를 최소 범위로 찾는다.
3. 필요한 파일만 확인한다.
4. 수정 계획을 짧게 작성한다.
5. 최소 범위로 코드를 수정한다.
6. 빌드 또는 실행 확인이 필요하면 필요한 명령어만 실행한다.
7. 변경 파일과 확인 방법만 간단히 정리한다.

---

## 19. 예외 상황

* 보안 문제, 데이터 손실 가능성, 인증/결제 관련 문제는 토큰 절약보다 안정성을 우선한다.
* PWA 캐시, 서비스 워커, 라우팅, 인증, 결제, 데이터베이스 구조 변경은 영향 범위를 먼저 확인한다.
* 대규모 변경이 꼭 필요한 경우에는 먼저 이유와 변경 범위를 짧게 설명한다.
* 사용자의 요청이 모호해서 잘못 수정할 위험이 큰 경우에는 작업을 진행하기 전에 핵심 질문 1개만 한다.
