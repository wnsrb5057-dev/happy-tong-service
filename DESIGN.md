# 해피통서비스 디자인 방향 개정

해피통서비스의 디자인 콘셉트는 **Public Care Utility**다.

공공서비스의 신뢰감, 모바일 앱의 간결함, 실무 대시보드의 효율성을 결합한 지역 돌봄 운영 시스템을 목표로 한다.

이 문서는 기존 외부 레퍼런스 중심 문서를 해피통서비스 전용 기준으로 개정한 것이다. 특정 디자인 시스템을 그대로 복제하지 않고, 아래 참고 방향을 해피통서비스 맥락에 맞게 재해석한다.

## 참고 방향

- 기본 토대: KRDS, 대한민국 정부 디자인 시스템
- 체커 모바일 UX: Toss식 간결한 모바일 사용성
- 관리자 업무 화면: SmartHR식 실무형 B2B 대시보드

## 1. 핵심 디자인 원칙

### 1-1. 신뢰감

- 공공기관, 지자체, 복지기관이 사용해도 어색하지 않아야 한다.
- 과하게 화려한 그래픽보다 안정적이고 정돈된 UI를 우선한다.
- 시각적 강조는 색상 하나로 과도하게 밀어붙이지 않고, 구조와 위계로 먼저 해결한다.

### 1-2. 모바일 우선

- 체커는 모바일 전용 사용자다.
- 체커 화면은 360px~430px 모바일 폭에서 가장 편하게 보여야 한다.
- 버튼은 크고 명확해야 하며, 한 손으로 누르기 쉬워야 한다.
- 주요 행동은 첫 화면 상단에서 바로 보여야 한다.

### 1-3. 실무 효율성

- 관리자는 지자체 공무원 또는 복지기관 실무자다.
- 관리자 화면은 모바일에서도 사용 가능해야 하며, PC에서는 더 넓게 볼 수 있어야 한다.
- 중요한 정보는 대시보드 상단에서 바로 확인 가능해야 한다.

### 1-4. 정보 밀도 구분

- 체커 화면은 정보 밀도를 낮춘다.
- 관리자 화면은 필요한 정보량을 유지하되, 요약 카드와 우선순위 중심으로 정리한다.

### 1-5. 이상징후 우선

- 이상징후, 긴급 확인 필요, 미처리 상태는 다른 정보보다 먼저 보여야 한다.
- 다만 색상은 과하지 않게 사용하고, 배경은 연하게, 배지는 선명하게 구성한다.

## 2. 컬러 토큰

### Primary

- Primary Blue: `#256EF4`
- Primary Hover: `#0B50D0`
- Light Blue Surface: `#ECF2FE`

### Text

- Text Primary: `#1E2124`
- Text Secondary: `#464C53`
- Text Tertiary: `#8A949E`

### Surface / Border

- Border: `#E6E8EA`
- Background: `#F8FAFC`
- Surface: `#FFFFFF`

### Status

- Success: `#228738`
- Success Surface: `#EAF6EC`
- Warning: `#FFB114`
- Warning Surface: `#FFF7E0`
- Danger: `#DE3412`
- Danger Surface: `#FDEEEE`

### 사용 원칙

- 기본 강조색은 `Primary Blue` 하나를 중심으로 사용한다.
- 위험도와 이상징후는 배지와 연한 배경으로 구분한다.
- 빨간색은 꼭 필요한 경우에만 사용하고, 넓은 면적을 과도하게 채우지 않는다.

## 3. 타이포그래피

### Font Family

- 기본 폰트: `Pretendard`, 시스템 sans-serif
- fallback: `-apple-system, BlinkMacSystemFont, "Segoe UI", "Noto Sans KR", Arial, sans-serif`

### 권장 크기

- Page Title: `22px~24px`, `font-weight: 700`
- Section Title: `17px~19px`, `font-weight: 700`
- Card Title: `16px~17px`, `font-weight: 600~700`
- Body: `14px~15px`
- Caption: `12px~13px`

### 사용 원칙

- 제목은 굵고 명확하게 표시한다.
- 설명 문구는 짧고 읽기 쉽게 구성한다.
- 모바일에서 작은 글씨를 남발하지 않는다.
- 텍스트 위계는 색보다 크기와 weight 차이로 먼저 만든다.

## 4. 레이아웃

### 체커 화면

- 모바일 앱처럼 간결하게 구성한다.
- 하단 탭바는 유지한다.
- 주요 행동 버튼은 화면 상단에서 바로 보이게 한다.
- 카드 개수를 줄이고, 요약 정보는 하나의 카드 안에 압축한다.
- 기록 작성은 1분 안에 완료할 수 있도록 단순하게 만든다.

### 관리자 화면

- 모바일에서도 사용 가능해야 한다.
- PC에서는 너무 좁게 중앙에 몰리지 않도록 `max-width` 또는 responsive grid를 적용한다.
- 대시보드, 통계, 보고서 화면은 PC에서 2열 이상의 카드/그래프 배치를 허용한다.
- 실무 화면이므로 과도한 애니메이션은 사용하지 않는다.

## 5. 카드 스타일

### 체커 카드

- `border-radius: 14px~16px`
- `padding: 16px`
- 배경은 흰색 또는 아주 연한 상태 색상
- 그림자는 최소화하고 border를 중심으로 구성
- 위험도에 따라 카드 배경 또는 좌측 라인을 다르게 표시

### 관리자 카드

- `border-radius: 10px~12px`
- `padding: 16px~20px`
- border 중심
- 정보 밀도는 체커보다 높게 가능
- 상태 배지와 주요 수치를 명확히 표시

## 6. 버튼 스타일

### 체커 버튼

- Primary button height: `52px~56px`
- `border-radius: 14px~16px`
- 핵심 행동 버튼은 `Primary Blue` 사용
- 보조 버튼은 흰색 배경 + Border 또는 `Light Blue Surface` 사용

### 관리자 버튼

- Primary button height: `44px~48px`
- `border-radius: 8px~10px`
- 실무 화면에 맞게 안정적이고 과하지 않게 구성

## 7. 배지 스타일

### 위험도

- 정상: `Success`
- 주의: `Warning`
- 위험: `Danger`

### 이상징후 수준

- 없음: 회색 또는 `Success`
- 확인 필요: `Warning`
- 긴급 확인 필요: `Danger`

### 처리 상태

- 접수됨: `Primary Blue` 또는 회색
- 처리중: `Warning`
- 완료: `Success`

### 사용 원칙

- 배지는 작지만 선명해야 한다.
- 카드 안에서 우선순위를 빠르게 인식할 수 있어야 한다.
- 넓은 단색 면보다 작은 상태 신호로 사용하는 편이 좋다.

## 8. 체커 UX 원칙

체커는 "동네 기반 외부 확인자"다.

- 체커는 복지 전문가가 아니며, 사회복지사를 대체하지 않는다.
- 독거노인의 생활 주변을 가볍게 확인하고 이상징후를 관리자에게 전달하는 역할이다.
- 방문 중심 표현보다 확인 중심 표현을 사용한다.
- 보호자 연락처는 표시하지 않는다.
- 담당자 연락처를 표시한다.
- 상세 주소는 외부 확인 수행에 필요한 범위에서 표시할 수 있다.
- 주요 버튼은 상단에 배치한다.
- 기록 작성 화면은 대상자 유형별로 간단한 3단 선택 UI를 사용한다.
- 사진 첨부와 음성 메모는 선택 보조 기능으로 표현한다.

## 9. 관리자 UX 원칙

관리자는 지자체 공무원 또는 복지기관 실무자다.

### 우선순위

1. 대시보드
2. 이상징후
3. 통계
4. 보고서

### 우선 노출 정보

- 긴급 확인 필요 이상징후
- 미처리 이상징후
- 오늘 확인 예정인데 미완료
- 위험 대상자
- 기록 보완 필요

관리자 대시보드는 "오늘 운영에 문제가 있는지"를 바로 알 수 있어야 한다.

## 10. 요약 카드 원칙

숫자 카드가 너무 많이 나열되지 않게 한다.

### 체커 대상자 현황

- 전체 `N명` · 오늘 `N명` · 주의/위험 `N명`

### 관리자 운영 현황

- 확인 예정 `N건` · 완료 `N건` · 미작성 `N건`
- 이상징후 `N건` · 긴급 확인 `N건`

### 대상자 현황

- 전체 `N명` · 오늘 확인 `N명`
- 정상 `N명` · 주의 `N명` · 위험 `N명`

## 11. 정보 공개 원칙

### 체커 화면

- 본인 담당 대상자만 표시
- 보호자 연락처는 숨김
- 담당자 연락처는 표시
- 민감한 건강 상세 정보는 최소화
- 외부 확인 수행에 필요한 범위에서만 주소 표시

### 관리자 화면

- 전체 대상자 표시 가능
- 보호자 연락처 표시 가능
- 담당 체커와 전체 기록 확인 가능
- 이상징후 처리 상태 변경 가능
- 통계와 보고서 접근 가능

## 12. 반응형 원칙

### 체커

- `360px~430px`를 1차 기준으로 본다.
- 하단 탭바를 유지한다.
- 버튼은 크게 유지한다.
- 기록 작성은 스크롤이 길어지더라도 선택 입력이 쉬워야 한다.

### 관리자

- 모바일에서도 사용 가능해야 한다.
- PC에서는 카드와 그래프가 2열 이상으로 배치될 수 있어야 한다.
- PC에서 화면이 과도하게 좁은 단일 컬럼으로 보이지 않게 한다.
- 하단 탭 구조는 유지하되, 콘텐츠 영역은 더 넓게 쓸 수 있어야 한다.

## 13. 모션 원칙

- 실무 화면이므로 모션은 최소화한다.
- 탭 전환, 상태 표시, 섹션 등장 정도의 짧은 전환만 허용한다.
- bounce, spring, 과도한 scale 애니메이션은 사용하지 않는다.
- 경고/위험 정보는 애니메이션보다 레이아웃 우선순위로 강조한다.

## 14. 금지 또는 주의할 점

- 체커 화면을 복지 전문가용 업무 시스템처럼 만들지 말 것
- 관리자 화면을 모바일 앱처럼만 만들지 말 것
- 너무 많은 카드와 숫자를 세로로 나열하지 말 것
- 빨간색을 과하게 사용하지 말 것
- 보호자 연락처를 체커 화면에 노출하지 말 것
- AI가 복지 판단을 대신하는 것처럼 표현하지 말 것
- "감시", "순찰"처럼 보일 수 있는 표현을 피할 것
- "방문" 표현은 꼭 필요한 활동 유형에서만 사용할 것

## 15. 구현 체크리스트

### 체커 화면

- 첫 화면 상단에 빠른 행동 버튼이 있는가
- 요약 정보가 한 카드 안에 압축되어 있는가
- 3단 선택 UI가 390px 폭에서도 눌리기 쉬운가
- 보호자 연락처가 노출되지 않는가
- 이상징후 버튼과 경고 배지가 과하지 않게 우선 노출되는가

### 관리자 화면

- 오늘 운영 현황과 우선 처리 필요가 상단에 있는가
- 이상징후 목록이 우선순위 기반으로 정렬되는가
- 통계가 숫자 카드 나열보다 그래프 중심으로 보이는가
- 보고서가 AI 자동판단이 아니라 검토용 초안으로 표현되는가

## 16. 현재 프로젝트 대비 점검 포인트

현재 구현에서 아래 방향을 우선 점검한다.

1. 기존 `#005df9` 중심 토큰을 새 `Primary Blue #256EF4` 체계로 교체할지, 또는 단계적으로 alias 처리할지 결정
2. 폰트가 현재 신한 계열 기준으로 남아 있으므로 `Pretendard` 중심으로 정리 필요
3. 체커 카드 radius와 버튼 height를 모바일 기준으로 조금 더 명확히 정리할 필요
4. 관리자 화면은 현재 폭이 다소 좁아, PC용 content width와 2열 이상 배치를 더 적극적으로 허용할 필요
5. 상태 배지와 요약 카드 색상은 현재보다 공공서비스 톤으로 한 단계 절제할 필요

## 17. 한 줄 기준

해피통서비스는 "과장된 복지 앱"이 아니라, **공공 돌봄 운영을 위한 믿을 수 있는 모바일 확인 도구**처럼 보여야 한다.
