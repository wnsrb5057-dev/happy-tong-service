# 해피통서비스 PWA 설치 및 알림 권한 온보딩 계획

## 1. 문서 목적

이 문서는 해피통서비스에서 PWA 설치와 Web Push 알림 권한 허용을 어떤 사용자 경험으로 안내할지 정리하는 **설계 문서**입니다.

이 문서의 목적:

- 해피통서비스에 맞는 PWA 설치 안내 흐름을 먼저 정리한다.
- Web Push 구현 전에 사용자 경험과 운영 정책을 먼저 확정한다.
- 체커와 관리자, 총관리자 각각에게 어떤 수준의 설치/알림 안내가 필요한지 구분한다.

중요:

- 이 문서는 구현 문서가 아니라 설계 문서입니다.
- 실제 코드, 서비스워커, API, UI 컴포넌트 구현은 포함하지 않습니다.

---

## 2. 서비스 사용자 특성 정리

해피통서비스는 일반 소비자용 웹사이트가 아니라, 기관에 등록된 관리자와 체커가 반복적으로 사용하는 업무형 서비스입니다.

### 체커

- 현장에서 매일 생활 확인 업무를 수행합니다.
- 오늘 확인 일정, 미작성 기록, 이상징후 보고와 직접 연결됩니다.
- 모바일 사용 비중이 높고, 즉시 확인성과 재접속 편의성이 중요합니다.
- PWA 설치와 알림 권한의 필요성이 매우 높습니다.

### 관리자

- 이상징후 접수, 미처리 현황, 재배정 필요 대상자 등을 확인합니다.
- PC 사용 비중도 높지만, 모바일 알림의 필요성도 큽니다.
- 특히 새 이상징후 접수 알림은 업무 대응 속도와 직접 연결됩니다.
- PWA 설치는 권장, 알림 허용은 강하게 권장 대상입니다.

### 총관리자

- 기관 전체 현황 확인이 중심입니다.
- 1차 알림 MVP 대상은 아닙니다.
- PWA 설치는 선택 또는 약한 권장 수준이 적절합니다.

---

## 3. 도입 정책

해피통서비스는 등록된 업무 사용자를 위한 서비스이므로, PWA 설치와 알림 허용은 단순 편의 기능이 아니라 **운영상 기본 옵션에 가까운 기능**으로 봅니다.

### 정책 방향

- 관리자와 체커에게는 PWA 설치를 기본 운영 전제로 둡니다.
- 다만 1차 MVP에서는 설치하지 않았다고 서비스 이용 자체를 차단하지는 않습니다.
- 대신 로그인 후 설치 여부와 알림 허용 상태를 확인하고, 필요한 안내를 반복적으로 보여줍니다.
- 향후 기관 운영 정책에 따라 체커 설치를 사실상 필수로 강화할 가능성은 열어둡니다.

### 정책 문장 예시

`1차 MVP에서는 PWA 설치와 알림 허용을 강하게 유도하되, 서비스 이용 자체를 차단하지는 않습니다.`

## 5. 체커 온보딩 UX

체커는 1차 MVP에서 설치/알림 안내의 우선순위가 가장 높습니다.

### 권장 흐름

1. 체커 로그인
2. `/checker/home` 진입 시 설치 상태 감지
3. 미설치라면 상단 안내 카드 표시
4. 설치 완료 후 알림 권한 안내 카드 표시
5. 알림 허용 후 “알림 수신 준비 완료” 상태 표시

### 체커용 핵심 안내 메시지

- `해피통서비스를 홈 화면에 추가해 주세요.`
- `오늘 확인 일정과 미작성 기록을 놓치지 않도록 알림을 받을 수 있습니다.`
- `알림을 허용하면 오늘 확인 기록을 작성하지 않았을 때 알려드립니다.`

### 체커용 버튼 후보

- `설치하기`
- `iPhone 설치 방법 보기`
- `알림 허용하기`
- `나중에 하기`

### UX 의도

- 체커는 업무 수행과 알림 효용이 직접 연결되므로, 설치와 알림 허용을 강하게 유도합니다.
- 다만 로그인 직후 브라우저 권한 팝업을 즉시 띄우지는 않습니다.
- 먼저 “왜 필요한지”를 설명한 뒤 사용자가 버튼을 눌렀을 때 권한 요청이 시작되도록 합니다.

---

## 6. 관리자 온보딩 UX

관리자는 알림 필요성이 높지만, 체커보다 다양한 기기와 환경에서 접속할 수 있습니다.

### 권장 흐름

1. 관리자 로그인
2. `/admin/dashboard` 또는 `/admin/emergencies` 상단에 알림 안내 카드 표시
3. 모바일 사용자는 설치 안내 + 알림 허용 안내
4. PC 사용자는 브라우저 알림 허용 안내를 우선 표시

### 관리자용 핵심 안내 메시지

- `새 이상징후 접수 알림을 받을 수 있습니다.`
- `새 이상징후가 접수되면 바로 확인할 수 있도록 알림을 허용해 주세요.`
- `기관 운영자는 모바일에 해피통서비스를 설치해 두면 긴급 상황 확인이 더 쉬워집니다.`

### UX 의도

- 관리자에게는 “새 이상징후 대응”이라는 명확한 업무 가치를 먼저 보여줍니다.
- PWA 설치는 권장이지만, PC 브라우저 알림도 충분히 의미가 있으므로 안내 강도를 체커보다 약간 유연하게 둘 수 있습니다.

---

## 7. 총관리자 UX

총관리자는 1차 알림 MVP의 직접 대상이 아닙니다.

### 권장 방향

- 1차에서는 총관리자용 알림 온보딩은 우선순위에서 제외합니다.
- 필요하다면 약한 설치 안내만 보여주거나, 아예 생략할 수 있습니다.
- 향후 기관 전체 요약 알림이 필요해지면 2차에서 별도 검토합니다.

---

## 8. iOS 안내 정책

iOS는 일반 브라우저 접속과 설치형 PWA의 차이를 분명히 설명해야 합니다.

### 핵심 원칙

- iOS Safari 일반 접속만으로는 Web Push 동작이 제한적일 수 있습니다.
- 따라서 iPhone 사용자는 먼저 홈 화면에 추가하는 과정을 이해해야 합니다.
- 설치 후에 알림 허용 단계를 안내해야 합니다.

### iOS 안내 문구 예시

`iPhone에서는 Safari 공유 버튼을 누른 뒤 ‘홈 화면에 추가’를 선택해 주세요. 설치 후에는 알림을 허용하면 확인 일정과 이상징후 알림을 받을 수 있습니다.`

### UX 메모

- iOS는 설치 안내를 별도 단계로 분리하는 것이 좋습니다.
- “지금 알림 허용”보다 “설치 방법 보기”가 먼저 나와야 혼란이 적습니다.

---

## 9. Android 안내 정책

Android Chrome은 설치와 알림 흐름이 상대적으로 자연스럽습니다.

### 핵심 원칙

- 설치 프롬프트가 가능한 환경이라면 `설치하기` 버튼을 직접 연결할 수 있습니다.
- 설치하지 않은 상태에서도 브라우저 알림이 가능할 수 있지만, 업무형 서비스 특성상 설치를 우선 유도합니다.

### Android 안내 문구 예시

`Android에서는 ‘설치하기’를 누르면 홈 화면에서 해피통서비스를 바로 실행할 수 있습니다.`

## 10. 인앱 알림과 시스템 푸시 알림 구분

해피통서비스에서는 “화면 안 안내”와 “OS/브라우저 푸시 알림”을 분리해서 설명해야 합니다.

### 인앱 알림

- 사용자가 서비스 화면 안에 들어와 있을 때만 보이는 안내입니다.
- 배너, 안내 카드, 상태 텍스트, 설정 유도 UI가 여기에 해당합니다.
- 비설치 사용자에게도 보여줄 수 있습니다.

### 시스템 푸시 알림

- 브라우저 밖이나 백그라운드 상태에서도 표시될 수 있는 알림입니다.
- PWA 설치, 알림 권한 허용, push subscription 저장이 필요합니다.
- 업무상 중요한 알림은 이 시스템 푸시를 목표로 합니다.

### 1차 MVP 방향

- 1차 목표는 업무상 중요한 알림을 시스템 푸시로 보내는 것입니다.
- 비설치 사용자에게는 먼저 인앱 안내를 통해 설치와 권한 허용을 유도합니다.

## 11. 1차 MVP에서 구현할 온보딩 위치

### 체커

- `/checker/home` 상단
- `/checker/targets` 상단 또는 하단 보조 안내
- 기록 작성 직전 또는 미작성 상태 안내 영역

### 관리자

- `/admin/dashboard` 상단
- `/admin/emergencies` 상단
- 추후 설정 메뉴가 생기면 알림 설정 영역 연결 가능

### 총관리자

- 1차에서는 제외 또는 약한 설치 안내 수준

## 12. 상태 감지에 필요한 프론트 정보

실제 구현 전, 어떤 상태를 감지해야 하는지만 정리합니다.

### 감지 후보

- service worker 지원 여부
- Notification API 지원 여부
- PushManager 지원 여부
- `display-mode: standalone` 여부
- iOS Safari 여부
- Android Chrome 여부
- `Notification.permission` 값
  - `default`
  - `granted`
  - `denied`
- 서버에 저장된 push subscription 존재 여부

### 상태 해석 예시

- 설치되지 않음 + 권한 미요청: 설치 안내 우선
- 설치됨 + 권한 미요청: 알림 안내 우선
- 권한 거부: 설정에서 다시 켜는 방법 안내
- 지원 불가 브라우저: 지원 브라우저 전환 안내

---

## 13. 운영 정책

### 기본 방향

- 체커 교육/초기 안내에서 PWA 설치를 함께 설명합니다.
- 기관 도입 시 관리자 계정 발급과 함께 설치 가이드를 제공합니다.
- 1차 MVP에서는 설치를 강제하지 않지만, 설치하지 않으면 푸시 알림을 받을 수 없음을 명확히 안내합니다.
- 2차에서는 기관 설정에 따라 체커 설치를 사실상 필수로 둘 수 있습니다.

### 운영 메시지 원칙

- 설치를 “앱처럼 더 편함” 수준으로만 설명하지 않습니다.
- `오늘 확인 일정`, `미작성 기록`, `이상징후 대응`처럼 실제 업무 가치 중심으로 설명합니다.

---

## 14. 알림 권한 요청 UX 원칙

반드시 지켜야 할 원칙:

- 첫 화면 진입 즉시 브라우저 권한 팝업을 띄우지 않습니다.
- 먼저 왜 알림이 필요한지 설명합니다.
- 사용자가 `알림 허용하기` 버튼을 눌렀을 때만 권한 요청을 시작합니다.
- 권한을 거부한 경우 반복적으로 괴롭히지 않습니다.
- 거부 상태에서는 브라우저 설정에서 다시 켜는 방법을 안내합니다.

### 권장 흐름

1. 설명 카드 표시
2. 사용자가 CTA 버튼 클릭
3. 브라우저 권한 요청
4. 결과에 따라 성공/거부/미지원 상태 안내

## 17. 결론

해피통서비스는 일반 소비자용 사이트가 아니라 반복 업무형 서비스이므로, PWA 설치와 알림 허용은 선택 기능이 아니라 운영 효율과 안전성을 높이는 핵심 보조 수단에 가깝습니다.

따라서 1차 MVP에서는 다음 방향이 적절합니다.

- 체커: 설치와 알림 허용을 가장 강하게 유도
- 관리자: 이상징후 대응 중심으로 설치/알림 허용 권장
- 총관리자: 1차에서는 우선순위 낮음

핵심 원칙은 다음과 같습니다.

- 설치하지 않았다고 서비스 이용을 막지는 않음
- 알림 권한은 설명 후 사용자가 직접 요청 시작
- 인앱 안내와 시스템 푸시를 명확히 구분
- iOS와 Android 흐름을 다르게 안내
