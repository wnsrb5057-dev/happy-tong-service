# 해피통서비스 PWA Web Push 알림 도입 계획

## 1. 문서 목적

이 문서는 해피통서비스에 PWA Web Push 알림을 도입하기 전에, 범위를 안전하게 나누고 구현 리스크를 줄이기 위한 1차 계획 문서입니다.

이번 문서의 목표는 다음과 같습니다.

- 지금 당장 전체 알림 기능을 구현하지 않고 MVP 범위를 먼저 확정한다.
- 프론트, 서버, DB, 운영 측면에서 필요한 준비 범위를 정리한다.
- 보안상 민감한 정보와 브라우저 노출 가능 정보를 구분한다.
- 개발자와 운영 담당자가 같은 기준으로 이후 작업 순서를 합의할 수 있게 한다.

중요:

- 이 문서는 설계 문서입니다.
- 실제 SQL, 실제 서비스워커 코드, 실제 API 구현은 포함하지 않습니다.
- 실제 DB 마이그레이션, RLS 적용, Vercel 함수 구현은 다음 단계에서 별도로 진행합니다.

---

## 2. 도입 목적

해피통서비스의 알림은 단순 편의 기능이 아니라 돌봄 운영 누락을 줄이기 위한 보조 수단입니다.

알림이 필요한 이유:

- 체커가 오늘 확인 기록을 남기지 못한 채 업무를 종료하는 상황을 줄여야 합니다.
- 이상징후가 접수되었는데 관리자가 늦게 확인하는 상황을 줄여야 합니다.
- 관리자 조치 지연은 실제 돌봄 공백으로 이어질 수 있으므로, 필요한 시점에 빠르게 인지할 장치가 필요합니다.
- 대시보드에 접속해야만 알 수 있는 중요한 이벤트는 푸시 알림으로 보완할 가치가 있습니다.

정리하면, PWA 알림은 다음 문제를 줄이기 위한 기능입니다.

- 생활 확인 누락
- 이상징후 대응 지연
- 관리자 조치 지연
- 운영상 중요한 이벤트의 늦은 인지

---

## 3. 1차 알림 MVP 범위

이번 1차 MVP에서는 아래 2개 알림만 우선 검토합니다.

| 대상 | 시점 | 목적 | 예시 |
| --- | --- | --- | --- |
| 관리자 | 새 이상징후 보고 접수 시 | 빠른 확인/조치 시작 | `[해피통서비스] 새 이상징후가 접수되었습니다. 정춘자 대상자 · 반복 통화 미응답 · 확인이 필요합니다.` |
| 체커 | 매일 오후 5시, 오늘 확인 필요 대상자 중 미작성 기록 존재 시 | 당일 기록 누락 방지 | `[해피통서비스] 오늘 확인 기록이 남아 있습니다. 오늘 확인 필요 대상자 2명 중 미작성 기록이 있습니다.` |

### 왜 이 2개만 먼저 하는가

- 실제 조치가 필요한 이벤트만 먼저 다룹니다.
- 사용자 피로도가 가장 낮고, 운영 효용이 큰 알림부터 검증할 수 있습니다.
- 관리자와 체커 각각 1개 시나리오만 먼저 구현하면 테스트 범위를 통제하기 쉽습니다.
- 알림 권한 허용률, 클릭률, 브라우저별 동작 차이를 작은 범위에서 먼저 확인할 수 있습니다.

### 왜 나머지 알림은 2차 이후로 미루는가

- 알림 종류가 많아지면 운영 피로도가 급격히 올라갑니다.
- 같은 이벤트를 화면과 알림에서 중복 노출하면 체감 가치가 떨어질 수 있습니다.
- 구독 저장, 권한 UX, 중복 발송 방지, 시간 기준 정책이 먼저 안정화되어야 합니다.
- 1차 MVP 성공 후 실제 사용 패턴을 보고 확장하는 편이 안전합니다.

---

## 4. 1차에서 제외할 알림

다음 알림은 1차 범위에서 제외합니다.

- 모든 대상자 상태 변경 알림
- 모든 보고서 생성 알림
- 관리자 대시보드 요약 알림
- 체커 배정 변경 알림
- 반복적인 위험도 변화 알림
- 확인기록 저장 완료 알림
- 로그인/접속 관련 일반 알림

제외 이유:

- 이벤트 수가 많아 알림 피로도를 높일 수 있습니다.
- 사용자가 바로 행동하지 않아도 되는 정보까지 푸시로 보내면 효과가 떨어집니다.
- 우선순위가 낮은 알림은 앱 내부 알림 또는 대시보드 요약으로 충분할 수 있습니다.

## 5. 사용자별 알림 시나리오

### 5-1. 관리자 알림 시나리오

#### 시나리오

- 체커가 이상징후 보고를 등록한다.
- 관리자 대상 구독 정보가 있는 경우 푸시 알림 발송 대상에 포함된다.
- 관리자는 알림을 클릭해 이상징후 화면 또는 상세 화면으로 이동한다.

#### 알림 제목/본문 예시

- 제목: `[해피통서비스] 새 이상징후가 접수되었습니다.`
- 본문: `정춘자 대상자 · 반복 통화 미응답 · 확인이 필요합니다.`

#### 알림 클릭 시 이동 후보

- 1순위: `/admin/emergencies`
- 2순위: 해당 이상징후 상세 화면

권장 방향:

- 1차는 목록 화면(`/admin/emergencies`) 이동이 더 안전합니다.
- 상세 이동은 URL 파라미터, 권한, 상태값, 로딩 안정성 검증 후 2차에서 강화할 수 있습니다.

---

### 5-2. 체커 알림 시나리오

#### 시나리오

- 매일 오후 5시 또는 기관 설정 시간에 배치/크론이 실행된다.
- 오늘 확인 필요 대상자 중 기록이 남아 있지 않은 체커를 조회한다.
- 해당 체커에게 하루 1회 리마인드 푸시를 보낸다.

#### 알림 제목/본문 예시

- 제목: `[해피통서비스] 오늘 확인 기록이 남아 있습니다.`
- 본문: `오늘 확인 필요 대상자 2명 중 미작성 기록이 있습니다.`

#### 알림 클릭 시 이동 후보

- 1순위: `/checker/home`
- 2순위: `/checker/targets`

권장 방향:

- 1차는 `/checker/home` 이동이 더 적합합니다.
- 체커 홈에서 오늘 일정과 기록 현황을 바로 확인할 수 있기 때문입니다.

## 6. 필요한 DB 설계 초안

### 6-1. `push_subscriptions` 테이블 초안

실제 SQL은 이 문서에 작성하지 않고, 컬럼 설계 수준으로만 정리합니다.

예상 컬럼:

| 컬럼 | 설명 |
| --- | --- |
| `id` | 구독 레코드 기본 키 |
| `user_id` | `public.users.id` 기준 사용자 식별자 |
| `auth_user_id` | Supabase Auth 사용자 식별자 |
| `role` | `admin`, `checker`, `super_admin` 등 역할 |
| `organization_id` | 기관 단위 필터링용 |
| `endpoint` | 브라우저 PushSubscription endpoint |
| `p256dh` | PushSubscription 공개키 |
| `auth` | PushSubscription auth 값 |
| `user_agent` | 브라우저/디바이스 식별 참고용 |
| `is_active` | 현재 유효 구독 여부 |
| `created_at` | 생성 시각 |
| `updated_at` | 수정 시각 |
| `last_used_at` | 최근 발송 또는 최근 사용 시각 |

### 6-2. 설계 메모

- `endpoint`, `p256dh`, `auth`는 브라우저 push subscription 저장용입니다.
- 서버는 `SUPABASE_SERVICE_ROLE_KEY`를 사용해 발송 대상 구독을 조회하는 구조를 우선 검토합니다.
- 중복 구독, 만료 구독, 브라우저 변경 등은 `is_active`, `updated_at`, `last_used_at` 기준으로 정리할 수 있습니다.
- RLS 적용 방향은 별도 문서에서 다루고, 이 문서에서는 설계 초안만 다룹니다.

## 7. 필요한 프론트 수정 범위

아래는 실제 구현이 아니라, 이후 작업 시 프론트에서 손대야 할 범위 정리입니다.

### 예상 수정 범위

- 서비스워커 등록
- 알림 권한 요청 버튼 또는 설정 영역
- PushSubscription 생성
- subscription을 Supabase 또는 서버 API로 저장
- 알림 클릭 시 특정 경로로 이동
- 사용자가 권한을 거부한 경우 반복적으로 재요청하지 않는 UX

### 프론트 설계 메모

- 알림 권한 요청은 로그인 직후 자동 팝업보다, 사용자가 맥락을 이해할 수 있는 설정/안내 영역에서 시작하는 것이 안전합니다.
- 체커와 관리자의 알림 목적이 다르므로 역할별 안내 문구가 필요할 수 있습니다.
- 권한을 거부한 사용자는 같은 세션에서 계속 요청하지 않아야 합니다.
- 브라우저가 푸시를 지원하지 않거나 PWA 설치가 필요한 경우, 대체 안내 문구가 필요합니다.

## 8. 필요한 서버/Vercel 함수 수정 범위

실제 구현은 하지 않고, 후보만 정리합니다.

### API/함수 후보

- `/api/push/subscribe`
  - 브라우저 subscription 저장
- `/api/push/send-emergency`
  - 새 이상징후 접수 시 관리자 알림 발송
- `/api/cron/checker-reminder`
  - 매일 특정 시각 체커 미작성 리마인드 실행
- `/api/push/unsubscribe`
  - 사용 중지 또는 만료 처리 보조

### 서버 설계 메모

- 실제 push 발송은 브라우저에 노출되지 않는 서버 환경에서만 처리해야 합니다.
- `VAPID_PRIVATE_KEY`, `SUPABASE_SERVICE_ROLE_KEY`, `CRON_SECRET`은 서버 전용입니다.
- 관리자 발송 로직과 체커 크론 로직은 분리하는 편이 운영상 추적이 쉽습니다.
- 중복 발송 방지와 실패 재시도 정책은 2차 세부 설계가 필요합니다.

## 9. 필요한 환경변수 설명

| 환경변수 | 용도 | 노출 가능 여부 | 비고 |
| --- | --- | --- | --- |
| `VITE_VAPID_PUBLIC_KEY` | 브라우저에서 PushSubscription 생성 시 사용하는 공개키 | 가능 | Vite 프론트에서 사용 가능 |
| `VAPID_PUBLIC_KEY` | 서버 측 공개키 참조용 | 서버 전용 권장 | 필요 시 서버 로직에서 사용 |
| `VAPID_PRIVATE_KEY` | 실제 Web Push 서명용 개인키 | 불가 | 절대 프론트 노출 금지 |
| `VAPID_SUBJECT` | VAPID 식별자 메일 주소 | 서버 전용 | 예: `mailto:...` |
| `SUPABASE_URL` | Supabase API 접근 주소 | 서버 사용 가능 | 프론트/서버 맥락 구분 필요 |
| `SUPABASE_SERVICE_ROLE_KEY` | 구독 조회/발송 대상 조회용 관리자 키 | 불가 | 절대 Vite 프론트 환경변수 금지 |
| `CRON_SECRET` | 크론 보호용 시크릿 | 불가 | Vercel Cron 보호용 |

### 보안 기준

- `VITE_VAPID_PUBLIC_KEY`만 브라우저 노출이 가능합니다.
- `VAPID_PRIVATE_KEY`, `SUPABASE_SERVICE_ROLE_KEY`, `CRON_SECRET`은 서버 전용입니다.
- `SUPABASE_SERVICE_ROLE_KEY`는 절대 Vite 프론트 환경변수로 넣으면 안 됩니다.

### `.env.local`과 Vercel 환경변수 구분

#### 로컬 개발에서 검토할 값

- `VITE_VAPID_PUBLIC_KEY`
- 서버 로컬 테스트가 필요할 경우 서버 전용 키들

#### Vercel Environment Variables에 넣을 값

- `VAPID_PUBLIC_KEY`
- `VAPID_PRIVATE_KEY`
- `VAPID_SUBJECT`
- `SUPABASE_URL`
- `SUPABASE_SERVICE_ROLE_KEY`
- `CRON_SECRET`

주의:

- 실제 값은 이 문서에 작성하지 않습니다.
- 환경변수 이름만 후보로 관리합니다.

## 10. 구현 단계 제안

1. 문서 작성
2. `push_subscriptions` 테이블 migration 초안 작성
3. 알림 권한 UI 설계
4. 서비스워커 등록
5. 구독 저장 API 구현
6. 관리자 이상징후 알림 1차 구현
7. 체커 미작성 리마인드 cron 구현
8. 테스트 계정으로 검증
9. 모바일 브라우저별 동작 확인
10. 운영 반영

### 권장 진행 순서 메모

- 먼저 관리자 이상징후 알림 단일 시나리오로 end-to-end를 검증하는 것이 좋습니다.
- 그 다음 체커 리마인드 크론을 붙이면 배치성 알림 이슈를 분리해서 볼 수 있습니다.

## 11. 브라우저/PWA 주의사항

- iOS Safari는 PWA 설치 상태에서만 Web Push가 제한적으로 동작할 수 있습니다.
- Android Chrome은 상대적으로 안정적으로 동작합니다.
- 브라우저 알림 권한을 한 번 거부하면 재요청 UX를 매우 신중하게 설계해야 합니다.
- 알림은 HTTPS 환경에서만 정상 동작합니다.
- 로컬 테스트와 Vercel 배포 테스트를 구분해야 합니다.
- 브라우저마다 백그라운드 동작, 절전 정책, 알림 표시 정책이 다를 수 있습니다.

### 테스트 우선순위 제안

- 1순위: Android Chrome + Vercel 배포 환경
- 2순위: 데스크톱 Chrome/Edge
- 3순위: iOS Safari 설치형 PWA

## 12. 알림 피로도 정책

초기 운영 원칙:

- 모든 이벤트를 알림으로 보내지 않습니다.
- 조치가 필요한 이벤트만 보냅니다.
- 체커 알림은 하루 1회 리마인드부터 시작합니다.
- 관리자 알림은 새 이상징후 중심으로 제한합니다.
- 같은 건에 대한 반복 알림은 추후 정책화합니다.

### 운영 관점 메모

- 체커에게 너무 자주 알림을 보내면 무시될 가능성이 커집니다.
- 관리자는 긴급도가 높은 이벤트부터 받아야 하며, 요약형 알림은 후순위가 적절합니다.
- 실제 발송 로그와 클릭률을 보고 2차 정책을 조정하는 것이 바람직합니다.

## 15. 결정 필요

아래 항목은 구현 전에 최종 합의가 필요합니다.

- 체커 리마인드 시간: 오후 5시로 할지
- 관리자 알림 대상: 기관 관리자 전체인지 담당 관리자 1명인지
- 알림 클릭 시 목록으로 보낼지 상세로 보낼지
- 미작성 기준: 오늘 확인 필요 대상자 중 기록 없음 기준인지
- 이상징후 알림 기준: 접수 시점 기준인지 미처리 상태 기준인지
- 사용자가 알림 수신을 끌 수 있게 할지

### 권장 초안

- 체커 리마인드 시간: 오후 5시 시작
- 관리자 알림 대상: 기관 관리자 전체보다 역할/기관 기준의 최소 대상 우선
- 클릭 이동: 1차는 목록 이동 우선
- 미작성 기준: 오늘 확인 필요 대상자 중 기록 없음
- 이상징후 알림 기준: 접수 시점 기준
- 수신 끄기: 1차에서는 브라우저 권한과 구독 비활성화 수준부터 검토

## 16. 결론

해피통서비스의 PWA Push 알림은 운영상 가치가 분명하지만, 한 번에 넓게 구현하면 권한 UX, 브라우저 호환성, 발송 정책, 보안 이슈가 동시에 커질 수 있습니다.

따라서 1차는 아래 두 알림만 MVP로 제한하는 것이 적절합니다.

- 관리자: 새 이상징후 접수 알림
- 체커: 오늘 확인 미작성 리마인드

이 문서를 기준으로 다음 단계에서는 실제 구현이 아니라 먼저 데이터 설계와 권한 UX 범위를 더 구체화하는 것이 안전합니다.
